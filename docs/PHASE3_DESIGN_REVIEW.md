## 2. CrossValidator 跨源交叉验证

### 2.1 背景与目标

**背景**: Phase 1 实现了单源数据采集，但缺乏跨源数据一致性验证。CrossValidator 用于验证来自不同数据源的同一品种数据是否一致。

**目标**:
- 验证多个数据源的数据一致性
- 自动检测和分类数据差异
- 提供冲突解决策略
- 生成数据质量报告

### 2.2 技术方案评估

#### 2.2.1 验证方法

| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **基于规则验证** | 简单、可解释 | 规则维护成本高 | 结构化数据 |
| **统计分析** | 自动化程度高 | 可能误报 | 大规模数据 |
| **记录链接** | 精确匹配 | 计算复杂 | 模糊匹配 |
| **数据对比** | 直观 | 性能差 | 小规模数据 |

**推荐方案**: 基于规则验证 + 统计分析的组合方案

#### 2.2.2 冲突解决策略

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **优先级策略** | 简单、可预测 | 需要预定义优先级 | 数据源有明确权威性 |
| **时间戳策略** | 自动化 | 依赖时间戳准确性 | 数据有可靠更新时间 |
| **投票策略** | 公平 | 需要多数源一致 | 多数据源场景 |
| **合并策略** | 保留更多信息 | 实现复杂 | 值可组合场景 |

**推荐方案**: 优先级策略（基于现有 SourceRegistry 的优先级）

## 3. 因子计算层

### 3.1 背景与目标

**背景**: 因子计算是量化投资的核心需求，Phase 1 预留了因子计算层骨架，但未实现。

**目标**:
- 设计可扩展的因子计算框架
- 实现常用技术因子和基本面因子
- 提供因子注册和管理机制
- 支持因子存储和查询

### 3.2 技术方案评估

#### 3.2.1 因子分类（基于行业标准）

**MSCI FaCS 标准分类**（8 个因子组，16 个因子特征）:

| 因子组 | 因子特征 | 数据需求 | 计算复杂度 |
|--------|----------|----------|------------|
| **价值** | 账面市值比、盈利收益率、股息收益率 | 财务数据 | 中 |
| **规模** | 市值、流通市值 | 行情数据 | 低 |
| **动量** | 价格动量、盈利动量 | 日线数据 | 低 |
| **波动率** | 历史波动率、特质波动率 | 日线数据 | 低 |
| **流动性** | 换手率、Amihud | 日线数据 | 低 |
| **质量** | ROE、毛利率、负债率 | 财务数据 | 中 |
| **成长** | 营收增长率、利润增长率 | 财务数据 | 中 |
| **杠杆** | 资产负债率、权益乘数 | 财务数据 | 中 |

**Fama-French 因子模型**:

| 模型 | 因子 | 命名约定 |
|------|------|----------|
| **三因子模型** | 市场超额收益、规模、价值 | Rm-Rf, SMB, HML |
| **五因子模型** | +盈利、投资 | +RMW, CMA |
| **扩展模型** | +动量、短期反转、长期反转 | +Mom, ST Rev, LT Rev |

**两大分类维度**:
- **风格因子**（Style Factors）: 解释单一资产类别内的风险和收益
- **宏观经济因子**（Macroeconomic Factors）: 解释跨资产类别的风险

#### 3.2.4 最新研究成果

**Microsoft Qlib**:
- 采用松耦合模块化架构，因子计算、模型训练、回测和执行分离为独立组件
- 通过 ExpressionCache 和 DatasetCache 双层缓存，将因子计算时间从 184.4 秒降至 7.4 秒（25 倍加速）
- 支持 Offline 和 Online 两种数据部署模式
- 提供专有 .bin 存储格式，优化金融数据科学计算
- 内置 Alpha158 和 Alpha360 因子集，覆盖中国和美国市场
- 支持公式化 Alpha 构建，使用表达式引擎（Ref、EMA、Mean、Rank 等算子）
- DataHandlerLP 支持可学习处理器（MinMaxNorm、ZscoreNorm、CSZScoreNorm 等）
- 标准接口模式：DataLoader → DataHandler → Dataset pipeline，基于 DataFrame I/O
- 支持声明式因子计算配置（YAML），包含工具过滤、时间范围、处理器链
- 支持自动每日数据更新（crontab），CSV/Parquet 到 .bin 转换管道

**璇玑XUANJI**:
- 采用七层架构（数据·因子·策略·回测·执行·风控·AI 自主调度）
- 使用 SQLite KV 状态总线作为模块间通信层
- 通过 PersistentRunner 模式管理 Python 常驻子进程

**Microsoft RD-Agent**:
- 首个数据驱动的多智能体量化因子挖掘框架
- 通过因子-模型联合优化实现约 2 倍年化收益率提升
- 使用 70% 更少的因子达到更好效果
- 自动化因子假设生成、实现和反馈循环

**AI-Quant-Factor-Miner**:
- 标准化因子函数接口：`function(df_raw, df_index) → DataFrame[SecuCode, TradingDay, FactorValue]`
- 代码文件 (.py) 与数据文件 (.parquet) 物理分离
- 使用 Parquet 存储因子数据

**StockTrader-AI**:
- 实现因子权重自进化机制：交易反馈→因子贡献分析→权重调整
- 向初始值回归 3% 防漂移，权重约束在 [5, 40] 范围内
- 支持牛市/震荡/熊市环境自适应

**Alphalens-reloaded**:
- Quantopian Alphalens 的维护版本，标准因子分析工具
- 提供四个核心分析 tear sheets：Returns、Information Coefficient、Turnover、Sector
- 与 Zipline-reloaded 和 Pyfolio 集成

**Zipline-reloaded**:
- Quantopian 回测引擎的维护版本，支持现代 Python/NumPy/Pandas
- v3.0+ 支持 Python 3.10-3.13、Pandas 2.0+、SQLAlchemy 2.0+、NumPy 2.0
- 由 Stefan Jansen 维护，用于《Machine Learning for Algorithmic Trading》一书

## 7. 结论与建议

### 7.1 关键结论

1. **CrossValidator**: 技术成熟，但实现复杂度高，建议分阶段实施
2. **因子计算层**: 需求明确，开源参考丰富，框架设计是关键
3. **更多数据源**: 集成成本低，可快速扩展数据覆盖

### 7.2 优先级建议

| 方向 | 建议优先级 | 理由 |
|------|------------|------|
| CrossValidator | P1 | 显著提高数据质量 |
| 因子计算层 | P2 | 满足量化投资核心需求 |
| 更多数据源 | P3 | 降低成本，提高覆盖度 |

### 7.3 实施建议

1. **CrossValidator**: 从小规模开始，先实现 2 个数据源的验证
2. **因子计算层**: 优先实现技术因子，基本面因子可后期扩展
3. **更多数据源**: 优先集成 EFinance（免费、易用）

### 7.4 下一步行动

1. **评审会议**: 组织开发评审会议，确认优先级
2. **详细设计**: 为优先级最高的方向编写详细设计文档
3. **原型验证**: 为核心功能开发原型，验证技术可行性
4. **资源确认**: 确认开发资源和预算
